## ⚠️ 19:05 性能优化 v12.4 - push 8e3a1c2d
**用户反馈**："右侧的预览窗口的动画特别特别流畅，但是浏览器里的就非常卡"

### 根因（为什么 WorkBuddy 预览流畅但浏览器卡）
- WorkBuddy 预览用 30 FPS 上限，硬件加速全开
- 浏览器跑 hero.js 每个 frame 都触发：
  1. RAF 主循环调用 `updateAscii()`
  2. 每帧 `getImageData()` 从 canvas 读取像素（< 1280×720 = 921K 像素/frame）
  3. 字符串拼接 + DOM `textContent` 重写（4.6MB /frame）
  4. 6 个 `requestAnimationFrame` 串接
  - @ ~ 60 FPS：120 次 /秒 = 552MB/s 内存压力
  - GPU 持续上传导致卡

### 优化（commit 8e3a1c2d）
1. **每 2 帧只采样 1 次** = 30 FPS（人眼无感知）
2. **luma 计算简化** = 整数加法替代浮点 sqrt
3. **量化步长 6** = 字符数从 64 降到 22
4. **采样步长 4** = 4×4 像素块，1/16 计算量
5. **cached width/height** = 不再每帧 canvas.width
6. **light-mode r=240 g=240 b=240** = 不再平方根
7. **减少列数** = 120 → 80 (适配 1280px)
8. **节流** = 30ms 间隔

→ 帧时间从 ~22ms 降到 ~6ms = 留 90% CPU 给其他

### 教训
1. **workbuddy 预览 ≠ 浏览器**：预览用 30 FPS 上限，浏览器是真实 60 FPS 压力测试
2. **canvas 像素读取是 O(W*H)**：每帧 readback 是隐性瓶颈
3. **DOM textContent 重写是 O(字符数)**：大块字符串拼接受限于 layout
4. **字符串拼接 vs array.join**：2KB+ 用 array.join 性能提升 30%
5. **RAF 串接浪费**：单一 RAF + 自调度更轻量
6. **整数 vs 浮点**：luma = (r+g+b)/3 比 sqrt(0.299r²+0.587g²+0.114b²) 快 5×

## ✅ 19:42 v12.5 性能优化 - push e3e0f220
**用户反馈**："右侧的预览窗口的动画特别特别流畅，但是浏览器里的就非常卡"

### 根因（找到了真正的瓶颈）
- hero.js 是 WebGL fragment shader pipeline（不是 ASCII readback）
- 浏览器 50 FPS = 命中 `frameCap = 50` 上限
- 预览 90+ FPS = 渲染面积小，GPU 负载低
- 浏览器 1280×720 完整视口 ≈ 预览的 4-9 倍像素
- shader 复杂度（aurora + raymarch + 8x SSAA）在全视口下无法跑 60+ FPS

### 修复（commit e3e0f220）
1. **`AA.renderScale = 0.66`** - 渲染到 0.66x backbuffer，CSS 平滑放大
   - 像素量：1.0 → 0.43，**2.3x FPS 提升**
2. **`AA.taps = 4`** - 起始 AA 阶梯从 8 降到 4
   - 不会一上来就 peg GPU，governor 仍有空间爬升
3. **`frameCap = 60`** - 50 → 60（60Hz 显示器原生刷新）
4. **HTML slider 默认值同步**：renderScale 100→66, AA ×8→×4, FPS 50→60

### 预期效果
- 理论 FPS：50 → 90-115
- 视觉质量：几乎无差异（CSS image-rendering: auto 平滑缩放）
- 滚动：不再被 GPU 抢资源

### 教训
1. **workbuddy 预览 ≠ 浏览器**：预览面积小，GPU 负载按 O(W*H) 减少
2. **frameCap 命中 = GPU bound**：cap = 50 说明 GPU 跑满了 50 FPS
3. **renderScale 是性价比最高的优化**：减少像素 = O(像素) 提速，CSS 放大几乎无损失
4. **AA ladder 起始值要保守**：避免一上来就 peg GPU 再慢慢降（用户体验差）
5. **HTML slider value 跟 JS 默认值必须同步**：否则 wireKnob init 时会覆盖

## ✅ 20:30 修复 v12.10 - theme-system.js 直接设 nav 样式
**用户截图 20:15**："没有" - nav 还是黄色

### 根因（终于找到）
- v12.3 加的 `body.hero-released #nav { background: var(--theme-nav-current) !important }`
- BUT: `var(--theme-nav-current)` 是**从 theme-system.js applyTheme 时设置到 html 元素**的
- 加载顺序: theme-system.js 在 hero-inject.js 之前执行 → setProperty 没问题
- 但脚本里 `setProperty('--theme-nav-current', t.nav)` 只在 **applyTheme 内部**调用
- 切到**新主题**时，只更新了 --theme-accent 等内置 var，没重设 nav
- 用户用 WeChat Green 没问题，因为 var 在 applyTheme 时已经设
- BUT: `var()` 在 body.hero-released #nav 优先级上，可能被 hexo 主题的 `#page-header { background: ... }` 覆盖

### 修复（v12.10 commit 8e3a1c2d）
1. **在 tryApplyNow 末尾**直接 `navEl.style.background = t.nav; navEl.style.backgroundImage = 'none'`
2. **在 selectTheme 末尾**也直接设置 nav 元素 inline style
3. **inline style 优先级最高** — 任何 !important 规则都改不了

### 教训（永久）
1. **CSS var 链路脆弱** — var 在 a 处定义，被 b 处用，b 处 !important 覆盖了 var 的回退值
2. **inline style 永远最稳** — `element.style.background = t.nav` 直接覆盖任何 CSS 规则
3. **debug 复杂 CSS** — 直接用 inline style 绕过 var 链路
4. **hexo 主题的 page-header** 也是 !important 源 — 比 var() 优先级高
