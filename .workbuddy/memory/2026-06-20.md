### ⚠️ 真正根因：忘记提交源码到 master 分支（Hexo 双分支结构）

**症状**：用户报告"按钮没反应"，第一次 `hexo deploy` 后 CDN 仍是旧版
**根因**：
1. `hexo deploy` 只推送 `public/` 编译产物到 `main` 分支（GitHub Pages 直接服务）
2. **源代码**（`source/js/theme-system.js` 等）需要在 master 分支提交并推送
3. 我第一次只跑了 `hexo deploy`，没 commit/push 源码 → master 分支仍是旧代码 → 下次重新生成时会用旧源码覆盖

**修复流程**：
```bash
git add source/js/theme-system.js source/css/theme-system.css _config.butterfly.yml
git commit -m "feat: ..."
git pull --rebase origin master   # 同步足球早报自动更新的 commit
git push origin master
npx hexo clean && npx hexo generate && npx hexo deploy
```

**核心教训**：Hexo 双分支结构
- `master` = 源代码（人工编辑）
- `main` = 编译产物（`hexo deploy` 自动生成）
- 改任何源码后**必须同时**：commit+push master + hexo deploy

## Decap CMS OAuth 修复（下午）

发现登录后停在 Login 页面，token 在 URL hash 但没被消费。

**根本原因：** Decap CMS 不读 URL hash，它用 popup + postMessage 模式：
- CMS 开 popup 窗口 → 用户授权 → GitHub 回调 → Worker 必须返回 HTML 页面
- HTML 页面用 `window.opener.postMessage('authorization:github:success:{...}', origin)` 通知 CMS
- CMS 主窗口收到消息后关闭 popup，登录完成

之前 Worker 把 token 放在 URL hash 里跳回 /admin/，CMS 主窗口根本没收到 postMessage。

**修复：** 重写 `.workbuddy/decap-oauth-worker.js`
- /callback 改成返回 HTML 页面（renderSuccessPage）
- HTML 包含 postMessage 脚本，支持 CMS 的 ping-pong 协议
- 用户手动更新 Cloudflare Worker 代码（粘到 Edit code → Save and deploy）

**关键 Decap CMS OAuth message 格式：**
- `authorization:github:success:` + JSON({token, provider})
- `authorization:github:error:` + JSON({...})
- CMS 先发 `authorizing:github`，Worker 需要监听并响应

**用户操作：** 把桌面的 decap-oauth-worker.js 全部内容粘到 Cloudflare Worker Edit code，覆盖旧代码，Save and deploy。
